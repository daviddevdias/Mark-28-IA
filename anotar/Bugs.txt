# Revisão Completa do Projeto J.A.R.V.I.S — MARK XXVIII

---

## Visão Geral do Sistema

O projeto é um assistente pessoal de IA chamado **J.A.R.V.I.S** (inspirado no famoso assistente do Homem de Ferro), construído em Python com interface gráfica via PyQt6/WebEngine. Ele combina reconhecimento de voz, síntese de fala, modelos de IA local (Ollama), integrações externas e automação do computador em um único sistema coeso. A arquitetura é modular, dividida em camadas bem definidas: interface, motor de IA, tarefas/ferramentas, armazenamento e integrações.

---

## Arquitetura Geral

### Ponto de Entrada — `main.py`

Este é o arquivo de inicialização do sistema. Ele:

- Define o `QApplication` do PyQt6 com contexto OpenGL compartilhado;
- Inicia o **Ollama** automaticamente se não estiver rodando, procurando o executável em caminhos padrão do Windows;
- Sobe a janela HUD (`JarvisUI`) e o painel de controle (`PainelCore`);
- Dispara a thread principal do motor (`engine_thread`) que roda um loop assíncrono separado;
- Registra os callbacks de fala, alarmes e monitor de voz antes de entrar no loop principal;
- Inicializa o bot do Telegram em uma thread daemon separada.

O loop principal (`engine`) fica aguardando comandos de voz continuamente: ouve, verifica wake word, processa e responde. Há tratamento de exceções no ciclo para evitar que um erro quebre o sistema inteiro.

**O que pode ser melhorado aqui:** a detecção do Ollama só funciona no Windows (usa `USERNAME` e `LOCALAPPDATA`). Para Linux e macOS, a lógica já tem `shutil.which("ollama")`, mas os caminhos alternativos são Windows-only. Seria interessante expandir para caminhos típicos do macOS (`/usr/local/bin/ollama`) e Linux (`/usr/bin/ollama`).

---

### Interface Gráfica — `painel.py`

O painel é uma janela `QMainWindow` que carrega um frontend HTML/JS via `QWebEngineView`. A comunicação entre Python e JavaScript acontece pelo `QWebChannel`, com um objeto `JarvisBridge` registrado como `"jarvis"` no canal.

**O que existe:**

- **JarvisBridge**: classe que expõe slots PyQt para o JS chamar. Inclui: executar comandos, salvar configurações, obter status de IA, listar microfones, testar voz, acionar análise visual, consultar clima, alternar modo IA, desligar sistema.
- **PainelCore**: janela principal com timers que atualizam CPU/RAM a cada 2 segundos e status da IA a cada 15 segundos.
- **Biblioteca de comandos**: função `montar_biblioteca_comandos()` que percorre as rotas registradas no `controller.py` e monta uma lista de comandos disponíveis para o painel exibir ao usuário.
- **Suporte a temas**: slot `obter_temas_sistema()` para o JS carregar temas visuais.
- **Notas**: integração com um arquivo `notas.json` acessível via configurações.
- **Hook de voz**: quando o Jarvis está falando, emite sinal para o frontend poder animar ou indicar visualmente o estado de fala.

**O que pode ser melhorado aqui:**

- O painel não tem sistema de **notificações toast** nativo no lado Python — tudo depende do frontend JS.
- Não há **atalho de teclado global** (hotkey global do sistema) para abrir/fechar o painel.
- A atualização de hardware poderia incluir temperatura da CPU, velocidade de rede e uso de disco, que já são coletados pelo `monitor.py` mas não chegam ao painel.
- Não existe mecanismo de **drag-and-drop** para arquivos na interface.

### Motor de IA — `ia_router.py` e `controller.py`

Estes dois arquivos formam o cérebro do sistema. Curiosamente, ambos declaram classes e variáveis quase idênticas (`IARRouter`, `Historico`, `SYSTEM`, `modelo`, `disponivel`, etc.), o que representa **duplicação de código**. Aparentemente `controller.py` era uma versão anterior ou alternativa do roteador, e ambos coexistem. Isso é o maior ponto de atenção estrutural do projeto.

**O que existe no roteador (`ia_router.py`):**

- Detecção automática de modelos Ollama com lista de preferência ordenada (llama3.2 > llama3.1 > qwen2.5 > mistral > etc.);
- Sistema de histórico de conversa com `deque` de tamanho máximo (20 turnos);
- Suporte a **tool calls** (ferramentas) apenas para modelos que suportam (qwen, phi3, mistral, llama3.1/3.2);
- Loop de despacho de ferramentas com no máximo 5 iterações por resposta;
- Montagem de conteúdo multimodal (texto + imagem base64) para análise visual;
- Cooldown de 30 segundos entre verificações de disponibilidade do Ollama;
- Ligação direta ao módulo de visão/monitor (`iniciar_monitor`, `parar_monitor`, `status_monitor`).

**O que existe no controller (`controller.py`):**

- Sistema de **rotas por palavra-chave** (`ROUTES`): lista de tuplas `(palavras, handler_async)` que mapeia frases normalizadas para funções assíncronas;
- Normalização de texto (remoção de acentos, lowercase);
- Expansão de prefixos (se o usuário diz "monit" entende "monitorar");
- Detecção de clima integrada antes de tentar rotas normais;
- Handlers para: silêncio, bloqueio de tela, minimizar, fechar janela, screenshot, limpar lixeira, modo trabalho, TV (ligar/desligar/YouTube), volume, Spotify (buscar/playlist/favoritas/pausar/continuar/próxima/anterior), YouTube, pesquisa web, monitoramento de tela, câmera, alarmes.

**O que pode ser melhorado:**

- **Eliminar a duplicação** entre `ia_router.py` e `controller.py` — unificar em um único arquivo;
- Adicionar suporte a **Gemini API** e **OpenAI API** como backends alternativos ao Ollama (a estrutura já prevê isso com `definir_modo()`, mas não implementa);
- O sistema de rotas é **case e ordem sensível** — rotas mais genéricas ("pesquisar") precisam vir depois das específicas ("pesquisar google"), o que já está correto mas pode se tornar problema ao adicionar novas rotas;
- Não há **intenção de contexto**: se o usuário diz "toca aquela música de antes", o sistema não tem como saber qual foi. Um sistema de contexto conversacional ajudaria;
- Falta suporte a **comandos compostos** ("abre o chrome e pesquisa python").

---

### Segurança de Comandos — `cmd_security.py`

Módulo bem estruturado com três camadas de proteção:

1. **Lista de bloqueios** com 20+ padrões regex para comandos destrutivos (`rm -rf /`, `mkfs`, `dd if=`, forks bombs, `wget | bash`, etc.);
2. **Detecção de injeção** via operadores de encadeamento (`;`, `&&`, `||`, `` ` ``, `$(`);
3. **Sistema de regras por categoria**: LEITURA, SISTEMA, REDE, DESTRUTIVO, BLOQUEADO — com flag de confirmação obrigatória para categorias destrutivas.

**O que pode ser melhorado:**

- Não há **log persistente** de comandos bloqueados ou executados — apenas log em memória;
- A detecção de injeção é baseada em string, não em parsing real do shell, o que pode ser bypassado com encoding;
- Comandos PowerShell complexos (com `-EncodedCommand`) não são cobertos;
- Falta **rate limiting** para evitar que a IA tente executar muitos comandos em sequência;
- Poderia ter um modo **auditoria** onde todos os comandos ficam registrados em arquivo.

---

### Cache de Ferramentas — `tool_cache.py`

Sistema de cache com TTL configurável por ferramenta. Usa hash SHA-256 dos argumentos como chave.

**Configurações de cache por ferramenta:**
- Clima: 600s (10 min)
- Busca web: 120s (2 min)
- Casa inteligente: 30s
- Outras (Spotify, apps, comandos): sem cache

**O que pode ser melhorado:**

- O cache é **in-memory** e se perde ao reiniciar — um cache persistente em SQLite ou JSON ajudaria para clima e buscas frequentes;
- Não há **invalidação automática** por evento (ex: ao trocar de cidade, o cache do clima deveria ser limpo);
- O limite de 256 entradas com LRU simples poderia ser substituído por uma política mais sofisticada;
- Falta **compressão** de valores grandes no cache (resultados de busca podem ser volumosos).

---

### Mapeador de Ferramentas — `tools_mapper.py`

Centraliza todos os handlers de ferramentas que a IA pode chamar via tool calls. Cada ferramenta tem um handler Python síncrono, e a função `despachar()` conecta ao sistema de cache.

**Ferramentas disponíveis:**
- `open_app` — abrir aplicativos;
- `computer_control` — controle do PC (volume, fechar, minimizar, print, bloquear, desligar, reiniciar);
- `web_search` — busca no Google via Playwright;
- `browser_control` — abrir URLs ou pesquisar;
- `youtube_video` — tocar vídeo no YouTube;
- `spotify_control` — controlar Spotify;
- `weather_report` — previsão do tempo;
- `set_reminder` — criar/listar/remover alarmes;
- `save_memory` — salvar dados na memória persistente;
- `agent_task` — criar planos com a IA;
- `code_helper` — gerar e opcionalmente executar código;
- `screen_analysis` — análise visual da tela;
- `smart_home` — controle da TV via SmartThings;
- `switch_ia_mode` — trocar modelo de IA;
- `cmd_control` — executar comandos de terminal com segurança;
- `cache_status` — consultar/limpar o cache.

**O que pode ser melhorado:**

- Falta ferramenta de **manipulação de arquivos** (criar, ler, mover arquivos do usuário com diálogo de confirmação);
- Falta ferramenta de **email** (enviar/ler emails via Gmail ou SMTP);
- Falta ferramenta de **calendário** (Google Calendar ou Outlook);
- Falta ferramenta de **WhatsApp** ou outras mensagens;
- O `gerenciador_codigo()` executa código gerado pela IA com avaliação de segurança, mas o sandbox é limitado — seria melhor usar subprocess em ambiente isolado ou Docker.

### Integração com Telegram — `telegram_bridge.py`

Bot completo do Telegram com 25+ comandos registrados, usando `python-telegram-bot` v20+.

**Comandos disponíveis:**
- `/jarvis <cmd>` — comando livre;
- `/status` — status do sistema;
- `/clima [cidade]` / `/amanha [cidade]` — previsão do tempo;
- `/alarme`, `/listar`, `/remover` — gerenciamento de alarmes;
- `/spotify`, `/pausar`, `/continuar`, `/proxima`, `/youtube` — música;
- `/abrir`, `/bloquear`, `/screenshot`, `/trabalho`, `/volume` — sistema;
- `/tvligar`, `/tvdesligar` — controle da TV;
- `/tela`, `/monitorar`, `/pararmonitor` — visão computacional;
- `/ia ollama|gemini|auto` — troca de modo IA;
- `/stop` — para a voz;
- `/ajuda` — lista de comandos.

**O que pode ser melhorado:**

- Não há **autenticação por chat_id** — qualquer pessoa que descubra o bot pode controlá-lo. Isso é um risco de segurança sério;
- Falta suporte a **mensagens de voz** (áudio → transcrição → processamento);
- Falta suporte a **imagens enviadas pelo Telegram** (enviar foto → analisar com visão);
- Não há **limite de taxa** (rate limit) por usuário;
- O bot não tem **estado de sessão** — não sabe quem enviou o último comando;
- Falta **modo inline** do Telegram para busca rápida.

---

### Áudio — `audio.py`

Sistema de áudio bem elaborado com vários cuidados:

- **TTS** via `edge-tts` (voz neural da Microsoft);
- **STT** via `speech_recognition` com Google Speech API;
- **Barge-in**: thread separada que ouve o microfone enquanto o Jarvis está falando — se detectar qualquer fala, interrompe a reprodução imediatamente;
- **Lock de I/O de áudio**: `threading.RLock` para evitar conflitos entre pygame e o microfone;
- Thread dedicada ao microfone (`JarvisMicLoop`) com fila de comandos/respostas para serializar as capturas;
- Suporte a índice de microfone configurável;
- Suspensão do pygame mixer antes de capturar para evitar eco.

**O que pode ser melhorado:**

- O STT usa Google Speech API online — sem internet, o reconhecimento falha. Adicionar **Whisper local** (OpenAI Whisper ou faster-whisper) como fallback seria muito valioso;
- A detecção de **wake word** está em outro módulo (`storage/wake.py` não incluído) — integrar Porcupine ou Vosk para detecção local de wake word melhoraria muito a responsividade e privacidade;
- O `edge-tts` também requer internet — adicionar `pyttsx3` ou `kokoro-tts` como fallback offline;
- Não há **ajuste automático de volume** — se o ambiente for barulhento, o reconhecimento piora sem feedback ao usuário;
- O arquivo de áudio TTS (`output.mp3`) é fixo, o que pode causar problemas se duas falas acontecerem simultaneamente — usar arquivo temporário com `tempfile` seria mais seguro.

---

### Alarmes — `alarm.py`

Sistema de alarmes persistente em JSON com capacidade de:

- Criar alarmes por hora (`HH:MM`) com data opcional (`YYYY-MM-DD`, `DD/MM`, `DD de mês`);
- Parser de linguagem natural para datas e horas em português;
- Disparar: TTS anunciando o alarme, música (`despertar.mp3`) em loop em canal separado do pygame, e ligar a TV via SmartThings;
- Parar o alarme via comando de voz;
- Alarmes repetitivos com flag `repetir`;
- Proteção contra duplo disparo no mesmo minuto com dicionário `ultimo_disparo`.

**O que pode ser melhorado:**

- Não há suporte a **alarmes semanais** (ex: "todo domingo às 8h");
- Falta suporte a **fuso horário** — usa `datetime.now()` sem tz;
- A música de despertar é hardcoded em `assets/despertar.mp3` — poderia ser configurável por alarme;
- Falta **snooze** (soneca): "adiar 10 minutos";
- Não há notificação visual além do áudio;
- Os alarmes concluídos ficam no JSON com `status: "concluido"` — sem limpeza periódica, o arquivo cresce indefinidamente.

---

### Browser — `browser.py`

Usa **Playwright** com Chromium em modo `headless=False` (janela visível) para:

- Busca no Google com digitação simulada caractere por caractere (anti-bot);
- Extração de snippet dos resultados;
- Tocar vídeos no YouTube;
- Suporte a stealth mode via `playwright_stealth` (opcional);
- Modo privado para buscas específicas;
- Sessão persistente com reconexão automática.

**O que pode ser melhorado:**

- Busca no Google é frágil — o layout do Google muda e os seletores CSS quebram;
- Falta integração com uma **API de busca real** (SerpAPI, Brave Search API, DuckDuckGo API) como alternativa mais confiável;
- O resultado extraído é texto puro sem estrutura — poderia retornar os top 3 links com snippets;
- Não há **download de arquivos** via browser;
- A sessão do Playwright abre uma janela visível (não headless), o que pode ser intrusivo.

---

### Controle do Computador — `computer_control.py`

Suporte multiplataforma (Windows, macOS, Linux) para:

- Fechar janela ativa;
- Minimizar todas as janelas;
- Captura de tela;
- Bloquear tela;
- Limpar lixeira;
- Ajustar volume do sistema;
- Desligar/reiniciar/cancelar desligamento.

**O que pode ser melhorado:**

- Ajuste de volume no Windows usa NirCmd (se disponível) ou PowerShell com SendKeys — ambos são frágeis. Usar a biblioteca `pycaw` seria mais robusto;
- Falta controle de **brilho da tela**;
- Falta **modo não perturbe** (silenciar notificações do sistema);
- Falta **gerenciamento de processos** (listar, matar processos por nome);
- Falta ação de **hibernar** além de desligar/reiniciar;
- O `bloquear_tela()` usa `cmd_security.avaliar()` mas os outros comandos não — deveria ser consistente.

---

### Monitor de Sistema — `monitor.py`

Sentinela proativa que verifica a cada 10 segundos:

- **Conexão com a internet** via socket para `8.8.8.8:53`;
- **Temperatura da CPU** via `psutil.sensors_temperatures()` com threshold crítico em 82°C;
- **Bateria** com alerta quando abaixo de 20% e descarregando.

Alertas são falados em voz pelo Jarvis.

**O que pode ser melhorado:**

- **RAM crítica** (acima de 90%) e **CPU crítica** (acima de 95%) são definidas como constantes mas não têm `checar_*` equivalente — o código tem as constantes mas não usa;
- Falta monitoramento de **uso de disco** (aviso quando partição principal estiver quase cheia);
- Falta monitoramento de **processos suspeitos** (CPU spike de processo desconhecido);
- O intervalo de 10 segundos é fixo — poderia ser configurável;
- Falta **log de histórico** dos alertas disparados com timestamps;
- Temperatura só funciona em Linux com `sensors_temperatures()` — no Windows, precisaria de `wmi` ou `openhardwaremonitor`.

---

### Abertura de Aplicativos — `open_app.py`

Mapeamento de aliases para nomes reais de aplicativos em Windows, macOS e Linux, com 35+ apps mapeados. Estratégias de abertura em cascata:

- Windows: `shutil.which` → `subprocess.Popen` → PowerShell via shell → `pyautogui` no menu Iniciar;
- macOS: `open -a` → PyAutoGUI no Spotlight;
- Linux: `shutil.which` → `xdg-open`.

**O que pode ser melhorado:**

- Falta integração com a **Microsoft Store** e **Winget** para apps não instalados;
- O fallback via PyAutoGUI no menu Iniciar pode ser bloqueado por políticas de empresa;
- Não há verificação se o app já está aberto (poderia focar a janela existente ao invés de abrir nova instância);
- Falta suporte a **argumentos** ao abrir apps (ex: "abre o VSCode na pasta X");
- A lista de aliases poderia ser **editável pelo usuário** via painel.

---

### Casa Inteligente — `smart_home.py`

Integração com **Samsung SmartThings** para controle da TV:

- Busca automática da TV por pontuação de relevância nos dispositivos;
- Suporte a ID manual de dispositivo;
- Cache de dispositivos com TTL de 60 segundos;
- Comandos: ligar, desligar, status, abrir YouTube;
- Detecção inteligente do candidato a TV com sistema de pontuação considerando labels, tipos e fabricante;
- Múltiplas tentativas com diferentes capacidades SmartThings.

**O que pode ser melhorado:**

- Só suporta **TV** — o sistema foi desenhado para mais dispositivos mas lâmpadas/tomadas foram desativadas nesta build. Reativar o suporte seria valioso;
- Falta controle de **volume**, **canal**, **input** (HDMI 1, HDMI 2) da TV;
- Falta integração com outros protocolos: **Tuya**, **Home Assistant**, **Matter/Thread**, **Google Home**;
- Não há suporte a **rotinas** (ex: "modo cinema" que liga a TV, escurece as luzes e fecha as persiana);
- Falta **descoberta automática** de novos dispositivos com notificação ao usuário.

---

### Spotify — `spotify_manager.py`

Singleton que gerencia a integração com Spotify via API oficial (`spotipy`) com fallback via interface gráfica (`pyautogui` + `pygetwindow`):

- Busca e toca música via API;
- Lista e toca playlists do usuário;
- Controla reprodução (play/pause/próxima/anterior) via API com fallback de teclado;
- Adiciona músicas aos favoritos;
- Foca a janela do Spotify automaticamente.

**O que pode ser melhorado:**

- Falta **exibir a música atual** ao Jarvis (ex: "O que está tocando?" → retornar nome/artista);
- Falta controle de **volume** do Spotify independente do volume do sistema;
- Falta **modo de repetição** e **shuffle**;
- Falta busca de **álbuns** e **artistas** (só busca tracks atualmente);
- O fallback via PyAutoGUI é frágil e depende do Spotify Desktop estar instalado;
- Falta **suporte a podcasts**;
- O cache de autenticação fica em disco — se corrompido, quebra silenciosamente.

---

### Clima — `weather.py`

Integração com **wttr.in** (serviço gratuito baseado no WorldWeatherOnline):

- Cache em memória com TTL de 10 minutos;
- Aliases para cidades brasileiras principais;
- Parser de cidade a partir do utterance do usuário;
- Previsão atual e previsão de chuva para amanhã;
- Retorno em JSON para o painel e em texto para TTS.

**O que pode ser melhorado:**

- wttr.in é um serviço de terceiros sem SLA — usar a **API oficial do OpenWeatherMap** ou **INMET** (para Brasil) seria mais confiável;
- Falta **previsão de 7 dias**;
- Falta **alertas meteorológicos** (tempestades, granizo);
- Falta integração com o monitor proativo (ex: se vai chover, alertar proativamente de manhã);
- O cache não é persistente — ao reiniciar, busca novamente.

---

### Webcam — `webcam.py`

Módulo simples que captura um frame da webcam via OpenCV e retorna em base64 JPEG com compressão configurável (qualidade 42, max 960px).

**O que pode ser melhorado:**

- Não há suporte a **múltiplas câmeras** com seleção pelo usuário;
- Falta **gravação de vídeo**;
- Falta **detecção de presença** (motion detection) para uso no monitor proativo;
- Falta **reconhecimento facial** para identificação do usuário;
- A qualidade 42 pode ser muito baixa para análise de código na tela — ajuste dinâmico seria melhor.

## O Que Pode Ser Adicionado ao Projeto

### 1. Reconhecimento de Voz Offline
Integrar **faster-whisper** ou **Vosk** para funcionar sem internet. A estrutura já suporta substituição do STT — basta criar um provider alternativo.

### 2. Wake Word Local
Usar **Porcupine** (Hey Porcupine, Hey Jarvis) ou **openWakeWord** para detecção local e eficiente da wake word, sem depender de reconhecimento contínuo do Google.

### 3. TTS Offline
Adicionar **Kokoro TTS**, **Coqui TTS** ou **pyttsx3** como fallback quando sem internet.

### 4. Multi-usuário com Reconhecimento de Voz
Identificar quem está falando (speaker diarization) e personalizar respostas por usuário.

### 5. Memória de Longo Prazo com RAG
Implementar um banco vetorial (ChromaDB, FAISS) para que o Jarvis recorde conversas antigas e documentos do usuário, usando RAG (Retrieval-Augmented Generation) nas respostas.

### 6. Autenticação no Telegram
Filtro por `chat_id` autorizado — sem isso, qualquer pessoa com o link do bot controla o computador.

### 7. Suporte a Gemini, OpenAI e Claude
O `definir_modo()` já existe mas retorna apenas "Ollama ativo". Implementar os clientes das outras APIs abriria muito mais capacidade, especialmente para análise visual avançada.

### 8. Interface Web Acessível Remotamente
Expor o painel via servidor HTTP local (Flask/FastAPI + WebSocket) para acesso pelo celular na mesma rede, sem precisar do Telegram.

### 9. Plugin de Email
Integração com Gmail API ou IMAP/SMTP para ler emails não lidos, responder e enviar mensagens.

### 10. Plugin de Calendário
Google Calendar API para criar eventos, listar compromissos do dia e receber lembretes com antecedência.

### 11. Modo Foco / Pomodoro
Timer de trabalho com bloqueio de distrações (fechar redes sociais, silenciar notificações, iniciar playlist de foco).

### 12. Dashboard de Produtividade
No painel, exibir: compromissos do dia, tarefas pendentes, tempo de trabalho, músicas mais ouvidas, histórico de comandos.

### 13. Integração com GitHub/GitLab
Consultar issues abertas, status de CI/CD, pull requests pendentes via voz.

### 14. Suporte a Home Assistant
Em vez de depender apenas do SmartThings, integrar com **Home Assistant** via REST API para controlar qualquer dispositivo smart home.

### 15. Histórico de Conversas Persistente
Salvar todas as conversas em SQLite com busca, permitindo ao usuário perguntar "o que eu pedi ontem de manhã?".

### 16. Agente Autônomo com Plano de Tarefas
Implementar um agente estilo **ReAct** que quebra tarefas complexas em passos, executa e verifica o resultado de cada passo antes de continuar.

### 17. Notificações Push no Celular
Usar **Pushover** ou **ntfy.sh** para enviar notificações para o celular quando alarmes dispararem ou alertas do sistema ocorrerem.

### 18. Suporte a WhatsApp
Via **Baileys** (Node.js) ou **whatsapp-web.js** com bridge Python para enviar e receber mensagens.

### 19. Controle de Brilho e Modo Escuro
Ajustar brilho do monitor e alternar tema escuro/claro do sistema operacional por voz.

### 20. Log Visual de Atividades
No painel, uma timeline de tudo que o Jarvis fez: comandos executados, alarmes disparados, alertas gerados, com possibilidade de desfazer ações.

## Pontos de Atenção Técnica (Bugs e Riscos)

1. **Duplicação entre `ia_router.py` e `controller.py`**: ambos têm `IARRouter`, `Historico`, `modelo`, `disponivel` e `detectar_modelo`. Isso pode causar estados inconsistentes. Verificar qual é o canônico e remover o outro.

2. **Sem autenticação no Telegram**: qualquer um que encontre o bot pode controlar o PC. Prioridade máxima de segurança.

3. **Arquivo de áudio TTS fixo (`output.mp3`)**: concorrência pode corromper o arquivo. Usar `tempfile.mkstemp()`.

4. **`asyncio.get_event_loop()` deprecated**: em Python 3.10+, `get_event_loop()` lança `DeprecationWarning` fora de contexto assíncrono. Usar `asyncio.get_running_loop()` onde aplicável.

5. **RAM e CPU críticas definidas mas não monitoradas**: as constantes `RAM_CRITICA` e `CPU_CRITICO` existem no `monitor.py` mas não têm função de verificação implementada.

6. **Alarmes sem fuso horário**: em sistemas com fuso diferente de Brasília, os alarmes podem disparar na hora errada.

7. **Cache sem persistência**: reiniciar o sistema invalida todos os caches, causando burst de requisições externas na inicialização.

8. **Playwright sem headless**: abre uma janela de Chrome visível ao buscar na web, o que pode ser intrusivo durante o trabalho.

9. **`tool_cache.py` `despachar()`** executa a função no thread pool com `run_in_executor`, mas muitas funções já são síncronas e bloqueantes — verificar se alguma função assíncrona está sendo chamada de forma incorreta.

10. **Dependência de `pygetwindow`**: só funciona no Windows — no Linux/macOS, o foco do Spotify via `gw.getAllWindows()` vai falhar silenciosamente.
